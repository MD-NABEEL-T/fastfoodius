## 🔴 CRITICAL ISSUES (Must Fix Before Production)

### 1. **RACE CONDITION: Stock Deduction Happens Twice**
**Severity:** CRITICAL  
**Location:** `orderController.js` - `createOrder()` (line ~44-54) and `verifyPayment()` (line ~149-174)

**Problem:**
- Stock is deducted in `createOrder()` BEFORE payment verification
- Stock is deducted AGAIN in `verifyPayment()` using Firebase transaction
- If Razorpay popup closes before payment completes → stock already deducted but payment failed
- If user cancels payment → stock lost permanently
- No rollback mechanism exists

**Flow:**
```
1. Student clicks "Confirm & Pay"
2. create-order API deducts stock immediately
3. Razorpay popup opens
4. User cancels popup
5. Stock is gone, but NO payment received ❌
```

**Fix:** Remove stock deduction from `createOrder()`. Only deduct in `verifyPayment()` AFTER signature verification succeeds.

---

### 2. **NO IDEMPOTENCY: Duplicate Orders from Retry**
**Severity:** CRITICAL  
**Location:** `orderController.js` - `verifyPayment()` (line ~195-210)

**Problem:**
- If frontend network error causes `verify-payment` to be called twice with same payment ID
- Backend will create TWO orders from ONE payment
- No unique constraint on `razorpay_payment_id`

**Flow:**
```
1. Payment succeeds: razorpay_payment_id = "rp_ABC123"
2. First verify-payment called → Order created ✓
3. Network timeout, frontend retries
4. Second verify-payment called → NEW order created ❌❌
5. Customer pays once, receives two orders
```

**Fix:** Store `razorpay_payment_id` in Firebase when creating order. Check if it exists before creating new order.

---

### 3. **NO AUTH MIDDLEWARE: UID Spoofing**
**Severity:** CRITICAL  
**Location:** `orderController.js` - `createOrder()` and `verifyPayment()`

**Problem:**
- Backend trusts `uid` from request body without verification
- Frontend's `auth.currentUser` could be spoofed if user manipulates request
- A student could place order as another user

**Example Attack:**
```javascript
// Attacker sends
{
  cart: [...],
  uid: "other-student-uid",  // NOT their own uid
  pickupSide: "..."
}
```

**Fix:** Extract uid from Firebase Admin SDK on backend, don't trust client-sent uid.

---

### 4. **TOKEN GENERATION RACE CONDITION**
**Severity:** HIGH  
**Location:** `orderController.js` - `verifyPayment()` (line ~176-185)

**Problem:**
- Token number generated by counting existing orders and adding 1
- If two payments complete in same millisecond, both could count same orders
- Both would generate token #5 (e.g.)
- No atomic increment

**Fix:** Use Firebase transaction to atomically increment a counter, or use unique timestamp-based token.

---

### 5. **STOCK VALIDATION BYPASS**
**Severity:** HIGH  
**Location:** `orderController.js` - `createOrder()` (line ~26-54)

**Problem:**
- No validation that price sent from frontend matches Firebase menu price
- Student could modify request to claim ₹100 item costs ₹1
- Qty verification exists (lines 37-40) but price is never validated

**Example Attack:**
```javascript
// Frontend sends actual: {price: 50}
// Attacker sends: {price: 10}
// Total becomes ₹10 instead of ₹50
```

**Fix:** On backend, fetch menu items and validate prices before calculating total.

---

### 6. **NO RAZORPAY EXPIRY HANDLING**
**Severity:** HIGH  
**Location:** `student.js` - payment handler (line ~540)

**Problem:**
- Razorpay orders expire after 30 minutes
- If user takes 35 minutes to pay, Razorpay order is invalid
- Stock was already deducted in `createOrder()`
- Payment fails, but stock never restored
- Customer loses inventory permanently

**Fix:** Implement timeout to cancel order and restore stock if payment not completed within 25 minutes.

## 🟠 HIGH-PRIORITY ISSUES

### 7. **Incomplete Item Processing in createOrder**
**Severity:** HIGH  
**Location:** `orderController.js` (line ~47)

**Problem:**
```javascript
if (!menuItem) {
  console.error(`[ERROR] Item not found: ${item.id}`);
  continue;  // ❌ Skips this item silently
}
```
- If item not found, function continues without error
- Order created with fewer items than customer ordered
- No error sent to frontend
- Stock may be partially deducted

**Fix:** Return error if ANY cart item is not found. Don't create order if validation fails.

---

### 8. **No Input Validation: Quantity Bounds**
**Severity:** HIGH  
**Location:** `orderController.js` - `createOrder()` and `student.js`

**Problem:**
- Frontend enforces 0 < qty ≤ 10, but backend doesn't validate
- Attacker could send qty: 1000
- Stock would be deducted incorrectly

**Fix:** Validate `0 < qty ≤ 10` on backend before processing.

---

### 9. **Shop Status Logic Complex & Client-Side**
**Severity:** MEDIUM  
**Location:** `student.js` (line ~24-67)

**Problem:**
- Shop opening/closing logic runs on CLIENT
- 60-minute grace period is client-side decision
- No server-side enforcement
- Inconsistency between two students viewing shop status simultaneously
- Client could manipulate `lastToggleTime` by faking time

**Fix:** Move shop status decision to server. Return `isShopOpen` boolean from backend API instead of having client calculate it.

---

### 10. **Firebase Concurrency Issue: update() + transaction() Mixing**
**Severity:** MEDIUM  
**Location:** `orderController.js`

**Problem:**
- `createOrder()` uses `await itemRef.update({stock: newStock})`
- `verifyPayment()` uses `await stockRef.transaction()`
- Mixed use of update() and transaction() can cause race conditions
- If two payments happen simultaneously, stock updates may interleave

**Fix:** Use transaction exclusively for all stock operations.

## 🟡 MEDIUM-PRIORITY ISSUES

### 11. **Platform Fee Hardcoded in Two Places**
**Severity:** MEDIUM  
**Location:** `orderController.js` (line ~63) and `student.js` (line ~454, 547)

**Problem:**
- Hardcoded ₹2 fee in three locations
- If fee needs to change, must update all three
- Easy to create mismatch (backend says ₹2, frontend displays ₹3)
- No single source of truth

**Fix:** Create `/api/get-config` endpoint returning `{platformFee: 2}`. Frontend fetches once on load.

---

### 12. **No Request Rate Limiting**
**Severity:** MEDIUM  
**Location:** `server.js`

**Problem:**
- No rate limiting on `/api/create-order` or `/api/verify-payment`
- Could be abused: attacker calls verify-payment 100 times per second
- Could flood Firebase with orders
- No DDoS protection

**Fix:** Add rate limiter middleware (e.g., express-rate-limit): 10 requests per minute per IP.

---

### 13. **Stock Restored Nowhere If Order Creation Fails**
**Severity:** MEDIUM  
**Location:** `orderController.js` (line ~195+)

**Problem:**
- Stock deducted in `createOrder()`
- If later operations fail (token generation, Firebase write), stock stays deducted
- No rollback mechanism
- Inventory stuck in inconsistent state

**Fix:** If any error after stock deduction, manually restore stock before returning error to client.

---

### 14. **Date-Based Token Generation Not Atomic**
**Severity:** MEDIUM  
**Location:** `orderController.js` (line ~176-186)

**Problem:**
- Token logic: count all orders today, add 1
- Uses `.once()` then calculates token
- Between `.once()` and order creation, another order might be created
- Both orders get same token number

**Fix:** Store current token counter in `shopConfig/currentToken` and use transaction to increment.

---

### 15. **No Error Message Sent on Stock Failure**
**Severity:** MEDIUM  
**Location:** `orderController.js` (line ~35-40)

**Problem:**
```javascript
if (currentStock < item.qty) {
  console.error(`[ERROR] Insufficient stock...`);
  continue;  // No error response!
}
```
- Frontend never told which item is out of stock
- Order created with fewer items

### 16. **No Firebase Realtime Listener Auth**
**Severity:** MEDIUM  
**Location:** `staff.js` and `student.js`

**Problem:**
- Direct Firebase listeners on `orders`, `menu`, `shopStatus`

   - Razorpay keys
   - Firebase config (partial)
   - Platform fee value

3. **HTTPS Only:** All traffic must be HTTPS in production
